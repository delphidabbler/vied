What are ... Fields & Macros

Fields and macros are probably the most useful component of the Version Information Editor and can be used to make updating of version information files much easier and less error prone.

This topic describes what fields and macros are and what is available. To learn how to use them see How do I use Fields & Macros.

Fields and macros are kinds of named variables which can be embedded anywhere in the text of a String Information item in a version information document. Macros can also be used in the File Version # and Product Version # Fixed File Information items.

When the document is exported as a resource file any fields and macros are replaced by the values they represent. Any trailing spaces in the resulting strings are trimmed.

Field and macro names are enclosed in angle brackets when embedded in text - e.g. <FIELDNAME> and <%MACRONAME>. From this it can be seen that the % symbol is use to differentiate macros from fields. Because the opening angle bracket is used to indicate the start of a field or macro, it can't be used directly in text. Instead the special field <<> must be used.

Not all fields are valid for use with each string information item. This restriction does not apply to macros.

While fields and macros are similar, they are not the same. The main difference is that there are a fixed number of fields, each of which has a pre-determined purpose, while macros are entirely user defined.

† All the date related fields refer to the date and time that the resource file is rendered.

Warning: Do not include any of the fields are replaced with the content of a string information item within the same item. For example don't use the <COMMENTS> field in the Comments item. Doing so causes a circular reference error to be reported. It is also possible to create a circular reference by defining string information fields in terms of each other. For example setting the following values in UI:

Comments = foo <FILEVERSION> bar
File Version = foo <INTERNALNAME> bar
Internal Name = foo <COMMENTS> bar

Causes a circular reference because:

  1. Parsing Comments requires replacing <FILEVERSION> by the contents of File Version, which requires:
  2. Parsing File Version requires replacing <INTERNALNAME> by the contents of Internal Name, which requires:
  3. Parsing Internal Name requires replacing <COMMENTS> by the conents of Comments, which takes us back to 1, and BOOM! we have a circular reference.

VIEd will trap such problems and will display an error message.

Macros

Whereas fields have predefined names and values, macros allow you to effectively create your own custom "fields" and assign values to them.

Macros are created and edited in the Macros dialogue box. Information about macros is stored with your version information data in a .vi file.

Macro names can be any number of characters long. They must begin with a letter or a number. Subsequence characters can be letters, numbers, a dash or an underscore. Examples of valid names are:

  • myMacro
  • Macro-1
  • M
  • 1My_Macro
  • 42

Examples of invalid macro names are:

  • -my-macro
  • _myMacro
  • my macro
  • $myMacro
  • #

There are three types of macro: Define, External and Include. Each is explained below.

Define

This is the most simple type of macro definition. It simply maps a macro name to a value. So, if Define macro myDef is defined as "Foo" then every occurence of <%myDef>is replaced by "Foo" when the version information resource is generated.

The purpose of this type of macro is to enable a single value that is used in multiple places to be defined in one place.

External

This type of macro definition references a file that whose contents contain the macro value, with certain restrictions, explained below. So if External macro myExt references file ".\value.dat" then whenever <%mExt> occurs it is replaced by the contents of ".\value.dat".

The restrictions mentioned above are:

  1. The file must exist. Relative file names are taken as being relative to the directory containing the .vi file. Absolute file names are permitted but not recommended.
  2. The value of the macro is set to the first non-blank line of the file. If there is no such line that the macro's value is an empty string.
  3. As a consequence of 1, you can't use (or view the value of) External macros that reference relative file paths until the the .vi file has been saved.

These macros are useful if there is a need to share the value recorded in the file in different parts of a program's source code.

Include

Like External macros, Include macros definitions reference data in a given file. Where Include macro definitions differ is that they include one or more additional macros that are defined in the file. The referenced file names both the macro names and their values.

To avoid names clashes between macros with the same name being defined in two or more Include files, the macros are referenced by the name assigned to the Include macro, followed by a dot and then the name of a macro from the included file.

For example, assume that macro alice is defined as ".\alice.dat" and macro bob is defined as ".\bob.dat". If alice.dat contains 

# Alice's Q & A

question=42
answer=56

and bob.dat contains 

# Bob's Q & A

question=Surname?
answer=Bloggs

Then we will have four new macros: <%alice.question>, <%alice.answer>, <%bob.question> and <%bob.answer>. When used, the macros are replaced with 42, 56, "Surname?" and "Bloggs" respectively. Notice that no macros named <%alice> or <%bob> are defined.

Restrictions are:

  1. Lines beginning with the # character are ignored and can be used as comments. Blank lines are also ignored.
  2. As with External macros you can't use (or view the value of) macros defined by Include macros that reference relative file paths until the .vi file has been saved.
  3. Like all macros, the names of those defined in Include files are restricted to alphabetic and numeric characters.

Note that you may include references to other fields in the value of a macro (but you can't reference other macros). This type of macro is useful whenever definitions in the form name=value are referenced in more than one source code file. For example:

Create a Define macro A with value Comments are <COMMENTS>
Set the Comments string information item to value Hello World
Set the Company Name string information item to value Macro A = <%A>

After resolving macros and fields the value of Company Name becomes Macro A = Comments are Hello World.

Warning: if you reference string information fields in your macros make sure you don't cause an circular references (see the discussion in the fields section above). Note that macros are replaced before fields are resolved.
